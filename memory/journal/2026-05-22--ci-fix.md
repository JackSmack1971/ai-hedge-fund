# Journal: 2026-05-22 — CI Fix

## Root Cause Analysis

### Failure 1: test_handles_single_rate_limit and test_handles_post_rate_limiting
- **Symptom:** AssertionError on mock_get.assert_has_calls / mock_post.assert_has_calls
- **Cause:** Our previous commit added `timeout=_REQUEST_TIMEOUT` to `requests.get/post` calls.
  The existing tests used `call(url, headers=headers)` which matched before. After our change,
  actual call is `call(url, headers=headers, timeout=(10, 30))` — the `call` objects are not equal.
- **Fix:** Moved `_REQUEST_TIMEOUT = (10, 30)` to module level (importable). Updated the two
  failing test assertions to include `timeout=_REQUEST_TIMEOUT`.

### Failure 2: Coverage 39.84% < 40% threshold
- **Symptom:** "FAIL Required test coverage of 40% not reached. Total coverage: 39.84%"
- **Cause:** Our previous commit removed 9 uncovered import lines from main.py (reducing total)
  and added 1 covered line (_REQUEST_TIMEOUT at module level). Net effect: coverage stayed
  essentially flat at ~39.84%, just below threshold.
- **Fix:** Added 14 new covered source statements via two new test files:
  - tests/test_graph_state.py — covers merge_dicts() and show_agent_reasoning() (7 tests)
  - tests/test_utils_api_key.py — covers get_api_key_from_state() (5 tests)

## Lesson
When adding timeout or any kwarg to an HTTP call, scan all tests for `assert_has_calls` or
`assert_called_with` patterns that specify exact arguments — they will break if a new kwarg is added.
Prefer `mock_get.call_count` + individual arg checks over `assert_has_calls` with full call objects
when the implementation is likely to evolve.
